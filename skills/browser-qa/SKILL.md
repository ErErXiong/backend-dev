# 浏览器 QA

用于后端改动存在浏览器可验收路径时。它不是替代单元测试或接口测试，而是在真实页面上确认用户能看到正确结果。

## 触发条件

- 新增功能、修复 Bug 或调整接口后，有页面入口可验证。
- 问题表现为页面不显示、按钮无效、列表/详情未刷新、表单提交异常、错误提示不对。
- 变更涉及请求参数、响应字段、权限状态、导入导出页面或前后端契约。

## 工作流

1. 先确认页面入口、账号状态、后端服务和前端服务启动方式；不清楚时从项目脚本、README、配置和现有测试中查。
2. 优先使用 Playwright；项目已有 e2e 测试时复用现有配置。新增 Playwright 用例时，`channel` 使用 `chrome`，断言采用增量方式，避免依赖残留状态。
3. 进入页面后等待动态内容稳定，例如 `networkidle`、关键请求完成或目标元素出现；不要只依赖固定 sleep。
4. 先侦察再操作：查看截图、DOM 可访问名称、console 错误和 network 请求/响应，确认真实失败点。
5. 执行最短用户路径，核对真实请求、状态码、请求体、响应字段、页面提示、列表/详情刷新结果。
6. 如果无法自动化浏览器验证，使用 Codex 浏览器、Chrome 或截图进行手工证据验证，并明确说明为什么没有 Playwright 验证。
7. 汇报时列出验证入口、浏览器方式、关键断言、失败证据或通过证据；不要在没有 fresh 验证输出时声称完成。

## 输出要求

- 验证方式：Playwright / Codex 浏览器 / Chrome 手工 / 未执行。
- 页面入口和用户路径。
- console 与 network 关键观察。
- 断言结果或截图说明。
- 未覆盖项和原因。
