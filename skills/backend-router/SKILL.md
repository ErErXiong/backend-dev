# 后端任务路由

先识别用户是否明确要求“只分析，不改代码”“按短流程处理”“按完整研发流程处理”或“先出方案，暂不实现”。显式要求优先于默认判断。

按以下优先级选路：

1. 出现线上、生产、紧急、数据修复、迁移、删除、权限或敏感配置时，先进入安全优先流程：说明影响范围、验证和回滚；未经确认不得发布、写生产数据或执行破坏性操作。
2. 出现禅道、工单链接或 Bug 编号时，使用已安装的 `zentao-bug-fix`；没有该技能时按普通 Bug 流程。
3. 出现报错、异常、失败、不生效、回归或修复时，使用已安装的 `bugfix-workflow`；先复现或收集证据，再做最小修复。
4. 出现方案、设计、需求不清、多个可行方案，或新增/修改业务行为且边界不清时，使用 `brainstorming`。只有架构、数据迁移、权限边界等高风险方案才使用 `grill-me`。
5. 页面不显示、按钮无效、列表/详情未刷新、表单提交异常、错误提示不对，或新增/修复后存在浏览器验收路径时，把 `backend-dev:browser-qa` 纳入验证；明确且小范围的后端改动使用 `backend-flow`，不要为了简单改动扩大为完整探索。
6. 新模块、跨前后端、多服务、数据库迁移或发布需求使用 `delivery-lifecycle`。

对导入、导出、Excel、模板、上传、请求参数或“前端未显示”任务，提醒后续流程同步检查字段契约、序列化/解析、接口调用和页面刷新链路；有页面入口时使用 `backend-dev:browser-qa` 做真实浏览器验证。

在说明路由时只用一行：`路由：<类型>；原因：<证据>；下一步：<动作>。` 无需让用户手工调用技能。
